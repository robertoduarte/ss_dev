Q3. What is the difference between GNU and GNUSH?

A3.    Very little - GNUSH is a version of the FSF GNU sources that has been prebuilt for your convenience.
The GNU Project was launched to develop a complete Unix-like operating system which is free software: the GNU system. GNU is a recursive acronym for "GNU's Not Unix" (it is pronounced "guh-NEW".)  The GNU system per se is not yet finished, but its offshoot GNU/Linux (which uses Linux as the kernel) is widely used.  See http://www.gnu.org/ for more information about GNU.

 

A subset of GNU is its development tools, which include command-line-basedcompiler, linker and various binary utilities. The source code for these tools,while freely available, generally does not include pre-built binaries. It has not been subjected to formal testing nor is there a guaranteed mechanism for new platform support. That solution is suitable to some developers, either because of cost constraints or because they want to evaluate GNU before embarking on anew software development project.

 

We endeavor to provide a readymade GNUSH tool chain based on the GNU standard for Renesas(formerly Hitachi) SH series of microprocessors / micro-controllers on Windows and on GNU/Linux. This tool chain is built on a standard reference base and delivered with pre-configured binaries. The GNUSH tool chain is subjected to thorough testing before it is released.  The advantage is that the embedded developers need not spend time in building and supporting the development tools and can concentrate on application development

    Q8. What is the difference between Cygwin and MingW ?

    A8. CygWin provides a transparent Unix emulation layer on Windows. This makes it possible to rebuild large, existing Unix applications with minimal changes to their source code, using GCC. The disadvantage is a substantial decrease in speed for both the compiler, and the windows applications built with it because of the overhead introduced by the Unix emulation layer (cygwin.dll)

    The Minimalist Gnu-Win32 project (MingW32) was started by Colin Peters. Instead of emulating Unix, it uses Microsoft's runtime DLLs(either CRTDLL.DLL or MSVCRT.DLL).

    MingW also contains the w32api package, which is a set of includes and import libraries to enable the use of Windows32 API.

    Q.11 When I generate .out file and load it into GDB, the initialization data does not appear in the .mdata section. Do I need to define the macro ROMSTART?
    OR
    When I make a ROM image, the .data section is not copied in ROM with the _mdata label, but only in RAM with the _data label. How can I get it copied to _mdata section?

    A.11 Every loadable or allocatable output section has two addresses. The first is the VMA, or virtual memory address. This is the address the section will have when the output file is run. The second is the LMA, or load memory address. This is the address at which the section will be loaded. In most cases the two addresses will be the same. An example of when they might be different is when a data section is loaded into ROM, and then copied into RAM when the program starts up (this technique is often used to initialize global variables in a ROM based system). In this case the ROM address would be the LMA, and the RAM address would be the VMA. The macro ROMSTART is used to copy initialized variables loaded at _mdata (LMA) to _data (VMA) at startup.

    If you are using debugger to load object file then ROMSTART should not be defined. The loader in debugger (GDB/HEW/HDI) loads the content at VMA so you don't have to copy it in the startup routine.

    If you are NOT using the debugger then make sure you have the following statements in the linker script
    .data : AT (_mdata) {
    _data = .;
    *(.data)
    _edata = .;
    } > ram

    This will load initialized variables at _mdata section in ROM.

    If you want to debug the code copying data in startup routine and are using debugger then loading S-Record file first and then COFF or ELF file will solve the problem.

    If you are using HEW, in Release configuration above thing is done by default. If you want to do it in Debug configuration, go to Options -> Linker -> Sections .data. Then in Modify section group -> Advanced, check "Reserve memory for section mapping" box and add label as _mdata.

    Q3. What are the SH related switches of gcc in GNUSH?

    A3.    As of now following switches are available...

    -m1

    Generate code for the SH1.

    -m2

    Generate code for the SH2.

    -m2e

    Generate code for the SH2e.

    -m3

    Generate code for the SH3.

    -m3e

    Generate code for the SH3e.

    -m4-nofpu

    Generate code for the SH4 without floating-point unit.

    -m4-single-only

    Generate code for the SH4 with a floating-point unit that only supports single-precision arithmetic.

    -m4-single

    Generate code for the SH4 assuming the floating-point unit is in single-precision mode by default.

    -m4

    Generate code for the SH4.

    -mb

    Compile code for the processor in big endian mode.

    -ml

    Compile code for the processor in little endian mode.

    -mdalign

    Align doubles at 64-bit boundaries. Note that this changes the calling conventions, and thus some functions from the standard C library will not work unless you recompile it first with `-mdalign'.

    -mrelax

    Shorten some address references at link time, when possible; uses the linker option `-relax'.

    -mbigtable

    Use 32-bit offsets in switch tables. The default is to use 16-bit offsets.

    -mfmovd

    Enable the use of the instruction fmovd.

    -mhitachi

    Comply with the calling conventions defined by Renesas(formerly Hitachi).

    -mnomacsave

    Mark the MAC register as call-clobbered, even if `-mhitachi' is given.

    -mieee

    Increase IEEE-compliance of floating-point code.

    -misize

    Dump instruction size and location in the assembly code.

    -mpadstruct

    This option is deprecated. It pads structures to multiple of 4 bytes, which is incompatible with the SH ABI.

    -mspace

    Optimize for space instead of speed. Implied by `-Os'.

    -mprefergot

    When generating position-independent code, emit function calls using the Global Offset Table instead of the Procedure Linkage Table.

    -musermode

    Generate a library function call to invalidate instruction cache entries, after fixing up a trampoline. This library function call doesn't assume it can write to the whole memory address space. This is the default when the target is sh-*-linux.

    Note : SH1 code is generated if no switch is given.

    Q1. How do I start writing a linker script?

    A1.    Following section explains the linker script commands for C,C++ programs in brief-

    The Entry Point

    ENTRY(symbol)

    First executable instruction in an output file e.g. ENTRY(_start)

             

    Memory Layout

    MEMORY

    {

                 name (attr) :ORIGIN = origin, LENGTH = len

                ...

    }

    The `MEMORY' command in linker script describes the location and size of blocks of memory in the target. 

     

    name- is a name used internally by the linker to refer to the region.

     

               (attr)- is an optional list of attributes

    e.g.`R' Read-only sections.

                    ‘W'  Read/write sections.

          `X' Sections containing executable code.

                  ...

    origin- is the start address of the region in physical memory.

    len-  is the size in bytes of the region

    e.g.

    MEMORY

    {

      ram (rw) : o = 0x0400000, l =512k

      rom (rx) : o = 0x0000000, l =512k

    }

    Section Definitions

    SECTIONS{ ...  secname : {    contents  }... }secnameis the name of the output section, and contents a specification of what goes there e.g. a list of input files or sections of input files

    Section Placement

    In a section definition, the contents of an output section can be specified by listing particular input files, by listing particular input-file sections, or by a combination of the two. An arbitrary data can also be placed in the section,and symbols can be defined relative to the beginning of the section.

    The section definition may include any of the following kinds of statement.

    filename

    e.g..data : { afile.o bfile.o cfile.o }

    filename(section )

    filename(section , section, ... )

    filename(section section ... )

    *(section)

    *(section, section, ...)

    *(section section ...)

     

    The Location Counter

    The special linker variable ‘.’ always contains the current output location counter.

     

    For the GNU C++ compiler, set up constructor and destructor tables. Set up the .tors section as follows:

    .tors:


    {

        __CTOR_LIST__ =.;

        LONG((__CTOR_END__ -__CTOR_LIST__) / 4 - 2)

        ___ctors = .;

       *(.ctors)

        ___ctors_end = .;

       LONG(0)

       __CTOR_END__ =.;

        __DTOR_LIST__ =.;

       LONG((__DTOR_END__ -__DTOR_LIST__) / 4 - 2)

        ___dtors = .;

       *(.dtors)

        ___dtors_end = .;

       LONG(0)

        __DTOR_END__ =.;

    } > rom


    The symbol __CTOR_LIST__marks the start of the global constructors and __CTOR_END__marks the end. Similarly the symbol __DTOR_LIST__ marks the start of the global destructors and __DTOR_END__ marks the end.The first word in the list is the number of entries, followed by the address of each constructor or destructor,followed by a zero word. GNU C++ will normally arrange to put the addresses of global constructors and destructors into the .ctorsand .dtorssections.

    The special section name ‘/DISCARD/’ may be used to discard input sections. 

    The comments in the linker script can be included just as in C, delimited by ‘/*’ and ‘*/’.

     The linker script also contains expressions.

    Expressions may inturn include integers, symbol names, operators, arithmetic functions.

               Some of the Section Data Expressions are –

               CREATE_OBJECT_SYMBOLS

                Create a symbol for each input file in the current section, set to the address of the first byte of data written from that input file.

               BYTE(expression)

                           Place one byte at the current address of that section.

               FILL(expression)

                           Specify the "fill pattern" for the current section.

               etc.

               Some of the Optional Section Attributes are -

           Start

    The output section can be forced to be loaded at a specified address.

               (NOLOAD)

               This directive will mark a section to not be loaded at run time.

               AT ( ldadr )

               ldadr specifies the load address of the section.

                >region

    Assign this section to a previously defined region of memory (e.g.
    ram,rom). etc.

     

    Option Commands

    The linker script may include a number of other commands that you can use for specialized purposes.

    OUTPUT_ARCH(bfdname)-

               Particular output machine architecture e.g.OUTPUT_ARCH(sh)

               STARTUP ( filename )

    Ensure that filename is the first input file used in the link process.

               Etc.

     

    SECTIONS - .text (code), .data (initialized data), .bss (uninitialized data). These 3 sections are present for all three object file formats -ELF, COFF, HMS.

    For ELF object file format you need to specify sections .init (invoke global constructors), .fini (invoke global destructors), .got (Global Offset Table),.rodata (Read Only Data) other than the 3 sections mentioned above.

     

    An example of .stack section definition :

    .stack0x043FC00 :

    {

       _stack = .;

       *(.stack)

    }>ram

     

    Depending upon the requirement of target the addresses of stack and text sections are to be specified.

     

    For the GNU C++ compiler, .gcc_exc section may also be needed for COFF and ELF if you get a warning for this section. This is related to the location of the main exception table for constructors.

    The .eh_frame section contains pointers that deal with exceptions.

     

    Refer to the linker scripts of sample tutorials. Refer to the Linker manual http://www.gnu.org/software/binutils/manual/ld-2.9.1/html_mono/ld.html for more information on linker script.

    Q2. What is a 'specs' file?

    A2. The ‘specs’ file contains the default options to be passed to the assembler/compiler/linker etc. Every GCC tool chain contains a specs file, named ‘specs’. This file is present in \lib\gcc-lib\\ directory.

    E.g. sh-elf tool chain is present in \lib\gcc-lib\sh-elff\3.2.1-GNUSH_v0301.

    If this file is removed/renamed then compiler, by default, uses builtin specifications. If not modified, the ‘specs’ file contains builtin specifications.

    This can be verified as follows:

    Execute command-

    $ sh-elf-gcc -v

    The first line of the output displays ‘Reading specs from ...’

    If the above command is given after renaming/removing the ‘specs’ file, the first line of the output displays ‘Using built-in specs’.

    The ‘specs’ file can be modified.

    Alternatively the builtin specifications can be overriden using -specs=’filename’ command line option. Multiple specification files can be specified on the command line and are processed in order from left to right.

    A lot of customisation can be done using the specs file. These changes are global and affect all users of the system.

    Further details about ‘specs’ language can be found in the source file gcc.c under heading ‘The Specs Language’.

    Q3. What is the difference between ELF, COFF and HMS? OR Which format, ELF or COFF is advisable for my applications?

    A3.The ELF, COFF and HMS are object file formats. The COFF and HMS file formats are identical. Please refer to FAQ of the General section.

    The ELF file format is more recent, contains more info and supports PIC.

    Please refer to the following links on the file formats:

    ELF -

    http://linux4u.jinr.ru/usoft/WWW/www_debian.org/Documentation/elf/elf.html
    COFF -

    http://www.delorie.com/djgpp/doc/coff/

    Regarding which format to use, its upto you, to decide which format suits your applications. Consider following points while choosing theobject format:

    ELF Format:

    1. Is more recent, contains more information.
    2. Supports relocatable code.
    3. Includes debug information. The format was designed to meet the debugging needs of different languages.
    4. Permits the section alignment to set on the section to section basis. But there is a high overhead attached to this feature. Please refer http://sources.redhat.com/ml/binutils/2000-01/msg00043.html

    COFF Format:

    1. It produces slightly compact output.
    2. Ability to debug is less, as compared to ELF format.
    3. The directory information is not included in the debugging output.
    Therefore while debugging the output and the source files are required in the same directory.

    Q4. My MAP file shows malloc functions though I have not allocated any data. Why?

    A4. The malloc functions are a result of the atexit() function called in the start-up file (normally crt1.o for sh-elf). Implementation of this function requires malloc and other associated functions. These calls are required for global constructor initialization of C++ programs.

    In 'C' applications, users can eliminate calls to malloc (and other nested calls) by writing their own start-up file. Using gcc's -nostartfiles option you can tell gcc not to use the default startup file and write your own startup code. For example refer start.s in Sample tutorials.

    In addition, COFF tool chain users, must call a dummy __main() function in their applications to eliminate the calls to malloc (and other nested calls).

    Q9. How can I customize low level functions printf, scanf for my project? OR How can I divert the standard output to UART for my project? OR How can I read the standard input from UART for my project?

    A9. Functions like printf,scanf internally calls low level functions write(), read() respectively. Low level routine write() is used by the printf to send the bytes to the output device. read() function is used by the scanf to read the bytes from the output device.

    To divert the characters towards UART you will have to modify write() function.

    This routine is available in syscalls.c file. This file is available in NEWLIB sources/newlib/libc/sys/sh. Once you change syscalls.c and link the file to your project, printf will call write() automatically.

    For e.g.

    int _write(file, ptr, len)

    int file;

    char *ptr;

    int len;

    {
    /* Your implementation to send the character to the serial port.*/

    // Call sci0_putc()

    return len;
    }

    To read the characters from SCI you will have to modify read() function.

    This routine is available in syscalls.c file. This file is available in NEWLIB sources/newlib/libc/sys/sh. Once you change syscalls.c and link the file to your project, scanf will call read() automatically.

    For e.g.

    int _read (int file,

    char *ptr,

    int len)

    {
    /* Your implementation to receive the character from the serial port.*/

    // Call sci0_getc();

    return len;
    }

    _read function reads only one character at a time therefore "len" is always 1.

    Q11. Can I manipulate the heap and stack?

    A11. Heap is the memory area where global allocation takes place. It starts after the _end symbol in the linker script and the end is the current stack pointer. Thus the location of the heap can be modified by changing the _end symbol in the linker script.The sbrk function in syscalls.c file (newlib\libc\sys\sh) in newlib sources has code to handle the stack and heap collision. By modifying the sbrk function one can manipulate the heap and stack size. To avoid rebuilding of the newlib sources make a copy of the syscalls.c file, modify it, and then link to the project.

    Q13. How can I use interrupts in my RAM based application which I want to debug using GDB stub ? OR Is there any way by which I can use interrupts in my application along with GDB stub interrupts ?

    A13. The vector table area resides in the user portion of the RAM area. Its start address must then be programmed into the Vector Base Register,
    VBR.

    Also to ensure that GDB stub will function correctly, the NMI, TRAPA, etc. interrupt vectors must be copied to the new user vector table before the value of VBR is changed.

    For instance if the user vector table was to reside at address 0x00401000, then the values for interrupt vectors at the following address would need to be copied across.

    Exception Vector

    Number

    Original Address

    New Address

    NMI

    11

    0x2C to 0x2F

    0x40102C to 0x40102F

    TRAPA

    32

    0x80 to 0x83

    0x401080 to 0x401083

    ERI1

    128

    0x200 to 0x203

    0x401200 to 0x401203

    RXI1

    129

    0x204 to 0x207

    0x401204 to 0x401207



    You need to write copyvect.s in your application like following,
    _CopyVect:

          mova vects,r0

          mov.l @r0,r1 ! Shadow vect tbl addr

           stc vbr,r2 ! Original vect tbl addr

           and #0,r0

           mov r0,r4 ! 0 in r4 and r0
    !trapa #32

           or #0x80,r0

           mov.l @(r0,r2),r3

           mov.l r3,@(r0,r1)

           mov r4,r0
    ! set VBR

           ldc r1,vbr

           rts

           nop

           .align 4
    vects:

           .long _vects

    Q2. What is a patch and how do I apply it ?

    A2. Patch is an output of diff command, which is to be applied to original file.

    Synopsis of diff is-

    diff [originalfile] [changedfile]

    The simplest way of applying a patch is to keep the original file and patchfile in the same directory.

    The synopsis of patch utility is -

    patch [options] [originalfile [patchfile]]

    but usually just give

    patch -pnum

    Where pnum means strip the smallest prefix containing num leading slashes from each file name found in the patch file. A sequence of one or more adjacent slashes is counted as a single slash.

    'Patch' normally replaces each file, file, with its new version, saving the original file in 'file.orig', and putting rejects hunks (if any) into 'file.rej'.

    If the patch is reversed or already applied you get the following message -

    1. Reversed (or previously applied) patch detected!

    Assume -R? [n]

    Apply anyway? [n]

    Result - This skips the patch.

    2. Reversed (or previously applied) patch detected!

    Assume -R? [n]

    Apply anyway? [n] y

    Result - The patch application fails.

    3. Assume -R? [n] y

    Result - The patch application succeeds.

    When 'y' is pressed, patch then attempts to
    swap each hunk around before applying it.

    For more information do

    $ man patch

    Q4. How do I install the GNUSH tool chain on a GNU/Linux machine?

    A4.    There is one RPM provided for installing GNUSH on Linux machine. An rpm is a specially formatted file which contains all the files, documentation and configuration files. For each object file format one binary rpm or package file is provided. The binary rpm filename is prefixed with i386.rpm and is used to install the package on the GNU/ Linux machine.

    (NOTE: To install the rpm on your machine you should have root access to the rpm database.)

    To install GNUSH tool chain on a GNU/Linux using RPM type the following on terminal window:

    rpm-ivh <package file>+

    e.g. rpm -ivh GNUSH_v0301_ELF-1-1.i386.rpm

    This will install GNUSH_v0301 ELF package if one is installing it for the first time. By default it installs it in /usr directory. To change the default directory use --prefix install mode option.

     To upgrade the package use the following command:

    rpm-Uvh <package_file>+

    The upgrade command can also be used even for the first install.

    To uninstall the package

    rpm-e <package_name>+

    e.g. rpm -e GNUSH_v0301_ELF

    Where GNUSH_v0301_ELF is the name of the package.

    To install or uninstall from X-Windows open the package from RPM-Manager and click on install or uninstall tab or click the rpm file in the file manager.

    For non-privileged users (i.e. other than root)

    You should have rights on the rpm database to install this rpm. The install and upgrade options to use are same as mentioned for privileged users.

    For more information refer man page of rpm or visit   http://www.rpm.org/max-rpm

    Q6. What should I do if I get an error ‘cannot exec `cc1': No such file or directory’?

    A6.     Error‘cannot exec cc1: No such file or directory’ will occur if cc1.exe is not present at its default location or in the path set in the PATH variable.

    Normally cc1.exe file is installed in e.g. ‘<Your Installation Folder>\lib\gcc-lib\sh-hms\3.2-GNUSH_v0204’. If, by any reasons, it is moved then the file location must be included in your PATH variable.

    This error may also occur if the environment variable GCC_EXEC_PREFIX is already set.

    If any other gcc version was installed earlier, that might have set this variable. This can be unset as GCC_EXEC_PREFIX= .
    Please refer to http://docs.freebsd.org/info/gcc/gcc.info.Environment_Variables.html for details of environments variables used by GCC.

    Q8. Are there any problems, if my host operating system is Windows 9X?

    A8. In Windows 9X, there is limit on number of characters in the command-line. This restriction on the length of the command line is of Operating system itself.
    This may cause problem, if any of the tool chain is installed in the directory which when accessed through command line exceeds the length limit.
    Issues & Workarounds:

    1. When the tool chain shortcut from 'Start' menu is invoked, the relevant batch file may not be found and Windows 9X may display error. To avoid this, copy the batch file associated with the
    shortcut to any directory on root. Modify the shortcut to refer to this file. In this way, the number of characters in the path of the batch file and in-turn in the command-line are reduced.
    2. In HEW (High-performance Embedded Workshop), this restriction may cause problem during linking. Please try the following steps in case you face the problem:

    a. Install HEW on the root directory.
    b. Run the linker command-line through the batch file.

    To perform this:
    - Disable the linker build phase for the project.
    - Build the project. This will compile the sources.
    - Create a batch file of the linker command line and save to the configuration directory.
    - Export the linker sections to a file and save the file to the configuration directory.
    - Remove the paths from the object files and add thelinker section file using -T option.
    - In the command prompt, run the batch file from configuration directory.
